# 論文を漫画にする AI — 必要なものとパイプライン

「Name（ネーム）哲学」に沿い、**まず視線の流れ（導入→対比→核→余韻）** を機械でも人でも共有できる **中間表現** を置くのがコツです。

## いま実装済み（MVP）

| 項目 | 内容 |
|------|------|
| API | `POST /api/ai/paper-outline` — 本文テキスト → ネーム案（JSON） |
| API | `POST /api/ai/pdf-text` — `multipart/form-data` の `file` に PDF（テキストレイヤー必須）→ 抽出テキスト |
| モック | `OPENAI_API_KEY` 未設定時はルールベースの `mock` 応答 |
| LLM | `OPENAI_API_KEY` 設定時は `OPENAI_MODEL`（既定 `gpt-4o-mini`）で JSON 生成 |
| フロント | トップの「お試し」で抄録貼り付け → API 呼び出し |

## 次に必要になるもの（順序目安）

### 1. 入力の質（PDF / LaTeX）

- **PDF テキスト抽出**: `pdfium` / `pdftotext` 等をサーバーで実行、または商用 OCR（スキャン PDF）。
- **著作権・プライバシー**: ユーザーアップロードの利用目的同意、保持期間、学内規程との整合。

### 2. パイプライン分割（非同期）

- 長文は **チャンク要約 → 全体合成** の 2 段にすると品質が安定しやすい。
- 重い処理は **キュー（SQS / Redis / pg job テーブル）** + ポーリング or WebSocket で UI に返す。

### 3. 構造化データ（コマ割りの前段）

- 出力スキーマを固定: `panels[]` に `{ "role": "...", "dialogue": "...", "visual_hint": "..." }` 等。
- 将来、**画像生成**や**テンプレコマ割**に流し込みやすくする。

### 4. 人間（漫画家）イン・ザ・ループ

- AI は **下書きネーム**まで。最終のコマ割・トーンは編集 UI で修正できるようにする（差別化）。

### 5. コスト・レート

- OpenAI 以外（自前モデル、国内 API）への差し替え口を `trait` / 環境変数で確保。
- **1ユーザーあたり回数制限**（既存の governor に加え、ユーザー単位クォータ）。

### 6. 評価

- 研究分野ごとの **ゴールデンセット**（入力論文 + 理想ネーム）で回帰テスト。

## 環境変数（API）

- `OPENAI_API_KEY` — 未設定でモック
- `OPENAI_MODEL` — 省略時 `gpt-4o-mini`

Fly 例:

```bash
flyctl secrets set OPENAI_API_KEY="sk-..."
# 任意:
flyctl secrets set OPENAI_MODEL="gpt-4o-mini"
```

## ナラティブ（データの流れ）

1. **読者の視線** — 抄録で「なぜ今それを読むか」を一文で掴む  
2. **対比** — 既研究との違いをコマで見せる  
3. **核** — 方法・結果を「絵になる」単位に分解  
4. **余韻** — 問いを残して次話（次の論文）へ誘う  

この 4 拍が同じ JSON に乗る限り、表現層（HTML リーダー・将来の画像）は差し替え可能です。
